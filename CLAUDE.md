# NACHTKAART

A map-first radar for electronic music events — club nights, raves, festivals — starting with the Netherlands. Open it on your phone and see what's on tonight around you: who's playing, where, until when, how far away. Built for the moment, not the plan.

## Product principles

- Mobile-first. Design for a phone screen; desktop is the adaptation, not the other way around.
- No ads, no algorithm, no feed. A utility, not a social network.
- Dense and utilitarian, like a timetable — never like a startup landing page.
- Default view: tonight, sorted by distance.
- Everything links out (RA, TicketSwap, venue sites, SoundCloud/Bandcamp/Spotify). We send traffic, we never trap it.

## Architecture (v1)

- Live at: https://nachtkaart.nl/ (GitHub Pages, custom domain via the repo's `CNAME` file — DNS is managed outside this repo)
- Python scraper → `data/events.json`, run nightly by GitHub Actions
- Static frontend on GitHub Pages: plain HTML/CSS/JS, no framework, no build step for v0
- Map: MapLibre GL JS with the CARTO Dark Matter basemap ("© OpenStreetMap" attribution is a license requirement — it always stays visible)
- Coverage: the scraper fetches RA area 176 ("All / Netherlands"), a national aggregate. `scraper/config.py` `SCRAPE_AREA_IDS` is a configurable list; `AREAS` documents verified NL area ids (Amsterdam 29, Rotterdam 174, Utrecht 175, Eindhoven 177, The Hague 178, Nijmegen 672).
- Data sources: Resident Advisor's unofficial GraphQL API first; djguide.nl later. Instagram: never.
  - RA's `Event` type also exposes a free-text `content` field (a promoter-written description, separate from the lineup). Confirmed to exist during the lineup-completeness check (2026-07-16) but not scraped or used anywhere yet — noted here so it isn't silently forgotten or silently added beyond what's been asked for.
  - RA's `Event.promoters { id name }` IS served to anonymous callers (verified 2026-07-19) — promoters are scraped and treated as first-class like artists (own pages). `Event.venue.id` is also scraped now (needed for stable venue-page URLs).
  - **No bulk historical events via `eventListings`**: querying an area with a *past* `listingDate` window returns placeholder junk (events dated 1980–1993), not real history. The only anonymous route to the past is per-entity `events(type: PREVIOUS)` — available on `artist`, `venue`, and `promoter`. Venue/promoter PREVIOUS is NL-scoped by nature (an NL venue's past events are in NL), unlike artist PREVIOUS which returns global gigs.
- Accounts: Supabase project (URL + publishable key live in `auth.js`, loaded on both pages before `app.js`/`artist.js`). Email magic-link auth; tables/RLS policies in `supabase/migrations/` (applied by hand via the Supabase dashboard's SQL Editor — no CLI, no DB password on this machine). Accounts are optional — everything works logged out.
- Later phase only: notification emails, personal iCal feed

## Data model

- `data/events.json` — the rolling live window (90 days ahead; `config.DEFAULT_DAYS_AHEAD`), overwritten each run.
- `data/archive/<year>.json` — permanent archive of every event ever seen, keyed by event id, one file per calendar year. Upserted on each scrape (changed fields updated, `first_seen`/`last_seen` tracked); events are never deleted.
- `data/venues.json` — hand-maintained venue → marker abbreviation/logo map, plus optional `lat`/`lng` overrides that always win over RA's own data. The scraper never writes it (only ever adds missing venues via `scraper/generate_venues.py`, never touches an existing entry). NB: this is the name-keyed marker map — distinct from the generated `data/venues/<id>.json` page files below.
- `data/artists/<id>.json` — one file per RA-linked artist (name, upcoming gigs from the live window, past gigs from the archive, socials). Generated by `scraper/artists.py` from events + archive on each scrape; consumed by `artist.html`.
- `data/venues/<id>.json` — one file per RA venue (name, city, coords, upcoming + past events). Generated by `scraper/venue_pages.py`, keyed by RA venue id; past gigs are bridged from the archive by venue *name* (archive rows predate the `venue.id` field) so backfilled history shows immediately. Consumed by `venue.html`.
- `data/promoters/<id>.json` — one file per RA promoter (name, upcoming + past events, `top_artists` they book most). Generated by `scraper/promoter_pages.py`; consumed by `promoter.html`. Promoter history is forward-only (the archive has no promoter data from before promoters became first-class) until a promoter backfill runs.
- `data/artist_socials.json` — real SoundCloud/Bandcamp/Discogs/website URLs per artist, populated by the backfill (below). Absent until a backfill has run.
- `data/search/` — the profile page's search index, generated from the archive by `scraper/search_index.py` each scrape: `<year>.json` compact past-event records (`{id, date, title, venue, venue_id, area, ra_url, terms}`, sorted by id so nightly git diffs stay minimal), `years.json` manifest, and deduped `venues.json`/`artists.json`/`promoters.json` name→id lists for the diary autocomplete. `profile.js` loads these (not the fat archive) to resolve WENT/past picks and to search for past parties.

## Venue coordinates

RA sometimes returns placeholder coordinates instead of a real location: exactly `(52, 5)` means "somewhere in the Netherlands, unknown," and `(0, 0)` ("null island") means the venue is genuinely TBA. `scraper/transform.py` detects both and nulls `venue.lat`/`lng`; the `(0,0)` case additionally sets `venue.location_tba: true` so the frontend can show "LOCATION TBA" instead of just quietly omitting a distance. `data/venues.json`'s optional `lat`/`lng` override (above) always wins over both RA's real data and this nulling. `scraper/geocode_venues.py` is a one-off, manual helper that proposes real coordinates for `(52,5)` venues via Nominatim (OpenStreetMap) — it only ever writes to `scraper/geocode_proposals.json` for review; a separate `--apply` run merges reviewed proposals into `venues.json`, never overwriting an existing hand-set override.

## Night logic

Electronic music runs past midnight, so grouping events by plain calendar date breaks exactly when it matters most: at 00:01, a party that started at 23:00 would flip to "yesterday" even though it's still on. Nachtkaart models **nights**, not dates — a core, deliberate rule, not an incidental detail:

> An event belongs to the night of `(start_time − 8h).date()`. "Right now" belongs to the night of `(now − 8h).date()`. Until 08:00 local, "tonight" still means the evening that began yesterday.

- **`scraper/nightlogic.py`** is the one canonical implementation (`night_of()` / `current_night()`). `scraper/transform.py` bakes each event's `date` field from its own `start` using this rule — not from RA's own `listingDate` bucketing, which doesn't reliably follow it. `scraper/artists.py` uses the same rule for its upcoming/past split, so artist pages agree with the main app.
- **`app.js`** mirrors the identical rule client-side (`nightOf()` / `currentNightAmsterdam()`), since "what night is it right now" is a live, viewer-clock-dependent question that can't be precomputed at scrape time. It drives the ticker, the stepper's default night, ON NOW's gating, and the TODAY/TOMORROW labels.
- Only new scrapes get the corrected `date` — already-archived historical events keep whatever `date` they were originally computed with; the archive was not retroactively recomputed.
- Events already finished, when viewing the current night: dimmed row (`--text-dim`) + `ENDED` tag, unfilled marker — but stay in their normal chronological slot, never hidden or resorted.

## Artist history / backfill

RA exposes each artist's full past history to anonymous callers via `artist(id).events(type: PREVIOUS)` (verified 2026-07-16), and their real socials on the same call. `scraper/backfill_artists.py` is a one-off, manual, long-running job that walks every known artist, seeds the archive with their past gigs, records socials, and rebuilds the artist files. It is resumable (per-artist responses cached) and is NOT part of the nightly CI scrape. Note: backfilling an artist pulls the full lineup of each of their past events, so the artist/archive sets cascade well beyond the seed list. **Until a backfill is run, artist history simply accumulates forward from the archive's start.**

## UI structure

- Ticker: one-line scrolling marquee of the selected night's headline events. Items are tappable (tap → select the marker + open the panel); the marquee pauses while a finger is down so the moving target is hittable.
- Filters: date, city, and FREE. Date uses **our own calendar dropdown** (`#calendar`, `renderCalendar()` in app.js) — a mono month grid clamped to [today .. last data date], primary on every platform (the native `<input type=date>` kept failing on iPhones and was removed). City is a dropdown built from distinct `venue.area` values (default label "ALL CITIES"); FREE is a toggle matching `price === "FREE"`. All three route through `eventsForDate()` in app.js (the single choke point feeding list/ticker/map) and persist in `localStorage` (`nachtkaart:cityFilter` / `:freeOnly`). When the city filter is ALL, each row carries an inverted city tag; a selected city suppresses it.
- Header/chrome: the city name sits under the NACHT/KAART logo (small, dim); top-right is PROFILE + LOG OUT (or LOG IN). The UPDATED timestamp lives at the end of the list. The control row (date/city/FREE) docks below the map and sticks to the top of the viewport when the list scrolls.
- Map: dark, square/crosshair markers; tap a marker → detail panel. Overlapping venues at the current zoom collapse into a single count-square (hand-rolled pixel-distance clustering, re-run on every `moveend`); tap it to zoom in and split it. The visitor's own position shows as an accent crosshair, only when geolocation is granted.
- Distance is shown either as km or as a rough travel-time estimate (KM / BIKE / WALK toggle in the detail panel, remembered in `localStorage["nachtkaart:distanceMode"]`, applied to list rows too). No routing API: straight-line distance × 1.3 detour factor at ~15 km/h cycling / ~5 km/h walking, `~`-prefixed as an estimate.
- List columns: Time | Event (Organizer) / Venue | Lineup | Tags
- Entity pages: `artist.html`, `venue.html` (name, city, a small map, favorite toggle, upcoming + past), `promoter.html` (upcoming + past, "often books" artists, FOLLOW placeholder). `profile.html` (logged-in only): FOLLOWING / VENUES / WANT TO GO / WENT. WENT is one chronology of archive `went` picks + private diary entries; "+ ADD PAST EVENT" is search-first (query the `data/search/` index → tap a result → it becomes a normal WENT pick) with a manual fallback that writes a private `custom_events` row (RLS own-rows, never on the public map/lists; linked names where an RA id is known, free text otherwise). Venue and promoter names across the app (list rows, detail panel, artist/venue/promoter gig rows) link to these pages when RA gives an id. List rows are a focusable `div[role=button]` (not a `<button>`) so those inner links are valid nested interactives — the row opens the panel except when a link is tapped.
- Detail panel (`panel.js`): one shared, self-contained module (injects its own markup + CSS, scoped to `#nkPanel`) used by every page — index (map/list), artist, venue, promoter, profile. Event rows everywhere open it (never jump straight to RA; the RA link lives inside). For entity/profile pages it resolves the full event by id (live window → the archive year for the gig's date), falling back to the row's own fields. Contents: sticky CLOSE bar, poster, date/time, price, distance (map page only), promoter(s), links out (RA + directions), lineup, favorite venue, went / want-to-go. `app.js` delegates to `window.NkPanel`.
- Distance: the panel's DISTANCE tile is a tap-to-cycle control — KM → WALK → BIKE → CAR (walk 5 / bike 15 / car 30 km/h, all ×1.3 detour), one `~`-prefixed value with km as a dim subline in time modes, times ≥60 min as "1H 11M". `NkPanel` owns the mode (persisted in `nachtkaart:distanceMode`) and the list rows read it.
- Follows carry a `kind` (`artist` | `promoter`); the profile FOLLOWING section splits into ARTISTS and PROMOTERS, and every profile list section is collapsible (count in the header, first ~10 + SHOW ALL, open/closed persisted). The explicit CLOSE control was reinstated after an earlier removal: at full height on a phone the panel filled the viewport, leaving no tap-outside area reachable, so scrim-tap alone wasn't enough. Tap-outside / swipe-down / Esc still all close it too.
